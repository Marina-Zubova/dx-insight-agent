## Friction Points

### 1. CI Security Gate Blocking Auto-Approval
- **Theme:** CI/CD  
- **Severity:** High  
- **Evidence:** On June 18, a bot comment explicitly states: _"This PR was **not** automatically approved for CI because the security gate failed"_ and lists possible reasons (50+ files, disallowed extensions, etc.). The author had to manually request slow test runs on June 27.  
- **Impact:** 9-day gap between PR creation and CI actually running the relevant slow tests. The security gate creates a bottleneck where CI simply doesn't run until a maintainer manually intervenes, and even then the author had to explicitly trigger `run-slow`.  
- **Recommendation:** Allow PR authors from trusted organizations (e.g., NVIDIA for their own model additions) to bypass the security gate, or implement a trusted-contributor allowlist. The gate should at minimum auto-notify maintainers for manual approval rather than silently blocking.  
- **Confidence:** High — directly evidenced in CI bot comments.

### 2. Modular Code Generation Gaps
- **Theme:** Tooling / Process  
- **Severity:** High  
- **Evidence:** ArthurZucker flagged: _"we don't do that usually (dependency on another model) — Let's probably use modular no?"_ The author responded: _"modular does not handle generation files... I'll merge with this and open a subsequent PR to add generation file handling in modular."_ This resulted in a follow-up PR (#46865) that refactored the processor away from modular.  
- **Impact:** The `modular` system — transformers' mechanism for avoiding code duplication between model files — doesn't support generation files, forcing contributors into workarounds. This creates technical debt and additional review rounds.  
- **Recommendation:** Extend the `modular` system to handle generation files, or document the limitation clearly so contributors know the expected pattern upfront.  
- **Confidence:** High — directly evidenced in review comments and follow-up PRs.

### 3. Stateful Design Rejected by Reviewer
- **Theme:** Code Review / Process  
- **Severity:** Medium  
- **Evidence:** ArthurZucker commented: _"arfff not a super fan of statefulness here! As its kinda hidden! I would make the next input in generate pass it instead."_ The author agreed and refactored, explaining it was due to `generate` not being modular enough.  
- **Impact:** Required an extra commit and discussion. The architectural feedback was valid but came late in the process.  
- **Recommendation:** Earlier architectural review for large feature additions could prevent rework. Consider a design doc or pre-review step for PRs above a certain size threshold.  
- **Confidence:** High — directly evidenced in review comments.

### 4. Reviewer Overload / Single Reviewer
- **Theme:** Communication  
- **Severity:** Medium  
- **Evidence:** Only one reviewer (ArthurZucker) provided substantive feedback on a 5,400+ line PR. The approval came 15 days after PR creation.  
- **Impact:** Single point of failure for review. If ArthurZucker were unavailable, this PR could have stalled indefinitely.  
- **Recommendation:** Require at least 2 reviewers for large PRs (configurable by lines changed or file count).  
- **Confidence:** Medium — inferred from review count; other maintainers may have been consulted offline.
